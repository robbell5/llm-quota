# llm-quota v1.1 Completion ‚Äî Display Preferences & Refresh Animation

**Date:** 2026-05-24
**Status:** Approved (design)
**Supersedes planning source:** `.planning/` is retired/dead context. This spec is authoritative for the remaining v1.1 work.

## Context

`llm-quota` is a small Go/Bubble Tea TUI that shows Claude Code and Codex
subscription quota usage (5-hour and 7-day windows, plus an optional Claude
Sonnet-only weekly window) in an always-running tmux pane. It reads local data
only ‚Äî a managed Claude cache file and Codex rollout JSONL ‚Äî with no network at
steady state.

The v1.1 milestone ("UI Polish and Small Features") spans three phases. Phase 7
(row alignment, Claude Sonnet limit, source freshness) is complete: it is
committed, builds clean, and passes `go test ./...`. The planning tool that
drove the earlier phases became unstable mid-milestone, so the remaining work is
being finished from this spec rather than from the dead planning folder.

This spec covers the two remaining phases:

- **Phase 8 ‚Äî Display Preferences:** bar style (segmented/solid) and provider
  visibility (Claude-only / Codex-only / both), set via CLI flags and toggled at
  runtime.
- **Phase 9 ‚Äî Refresh Animation:** subtle bar animation on load and on refresh,
  using the Bubbles progress component's built-in spring animation.

The intended outcome: a single user can configure the pane for their personal
setup and see refreshes happen through subtle motion, without the pane becoming
a noisy dashboard, with all behavior covered by deterministic tests.

## Locked Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Preference mechanism | CLI flags set initial state **+** runtime key toggles | User wants both set-once-at-launch and live adjustment. |
| Toggle keys | `b` = bar style, `v` = cycle visibility | Compact; only two new keys added to the existing `q`/`r`. |
| Animation engine | Bubbles built-in spring (`progress.Model`) | Idiomatic; provides easing/frame handling; minimal hand-rolled math. |
| Real-pane validation | **Automated tests only** | Accepted: real-tmux "feel" is a known, deliberately-accepted subjective gap; no human UAT step. |
| Bar styles | Segmented = `‚ñå` fill (current default); Solid = `‚ñà` fill | Maps directly to Bubbles `WithFillCharacters`. |
| Visibility model | 3-state enum `{both, claudeOnly, codexOnly}` | "Hide both" is structurally unrepresentable ‚Äî satisfies the no-empty-view requirement by construction. |

## Architecture

### Shared foundation: persistent per-row `progress.Model`

Today bars are stateless ‚Äî `renderProgressBar` (`internal/tui/view.go:272`)
builds a throwaway `progress.New(...).ViewAs(fraction)` on every render. Both
remaining features need the same change: a **persistent `progress.Model` per
quota row**, stored in the TUI `Model`.

- Phase 8 solid bars = set each model's `Full` rune (`‚ñå` ‚Üí `‚ñà`).
- Phase 9 animation = call `SetPercent` on that model and route `FrameMsg`.

Step 1 of implementation is therefore a behavior-preserving refactor: introduce
the persistent models and render through them with `ViewAs(currentFraction)`, so
existing tests stay green and no visible behavior changes yet. Both phases then
build on this seam.

Bars are keyed by the existing row identity (`quotaRowSpecs` index, or
`product`+`kind`). There are at most five rows: Claude 5h/7d, Sonnet 7d, Codex
5h/7d. Width is applied per render via `SetWidth(w)` based on the responsive
layout band; animation state (percent) persists across renders.

*Alternative considered and rejected:* keep bars stateless and make only
animated bars stateful. Rejected because it forks the render path and duplicates
the bar-styling logic across two code paths.

### Phase 8 ‚Äî Display Preferences

**State.** A new preferences value passed into the model:

```go
type BarStyle int   // BarSegmented (default), BarSolid
type Visibility int  // VisibilityBoth (default), VisibilityClaudeOnly, VisibilityCodexOnly

type DisplayPrefs struct {
    BarStyle   BarStyle
    Visibility Visibility
}
```

Threaded in via a new option `tui.WithDisplayPrefs(prefs)` (mirrors the existing
`WithReaders` / `WithClaudeHookInstalled` option pattern in
`internal/tui/model.go`). Defaults: segmented bars, both providers visible.

**CLI flags.** Parsed in `cmd/llm-quota/main.go` in the no-subcommand path
(before the TUI launches at `run()`'s `offerFirstLaunchInstall` / `StartTUI`
section). Flags apply only to the interactive TUI, never to the hook
subcommands.

- `--solid-bars` ‚Äî boolean; absence = segmented.
- `--only=claude` | `--only=codex` ‚Äî provider visibility; absence = both.
- `--help` / `-h` ‚Äî print usage and exit 0.
- Unknown flag or invalid `--only` value ‚Üí usage message on stderr, exit 2
  (matches the existing "unknown argument" convention at `main.go:62`).

Because `--only` is single-valued, no flag combination can hide both providers.

**Runtime toggles.** Added to the `tea.KeyPressMsg` switch in
`internal/tui/update.go` (which currently handles `q`/`ctrl+c` and `r`):

- `b` ‚Äî toggle bar style (segmented ‚Üî solid).
- `v` ‚Äî cycle visibility: both ‚Üí Claude-only ‚Üí Codex-only ‚Üí both.

**Apply in view.** In `renderRows` (`internal/tui/view.go:92`):

- Visibility filters which `quotaRowSpecs` rows render, **and** suppresses the
  freshness line and footer recovery hints for a hidden provider.
- Bar style sets the `Full` rune on the per-row `progress.Model`s.

**Docs (DOC-01).** README and `--help` document the flags and toggle keys
concisely.

**Bar style appearance:**

```
Segmented (current default)        Solid (--solid-bars / `b`)

```

### Phase 9 ‚Äî Refresh Animation

**Drive it.** In `internal/tui/update.go`:

- On first data for a row and on any refresh where the target percent changed,
  call `bar.SetPercent(target)` and collect the returned `tea.Cmd`s, batching
  them from `Update`.
- Initial load animates 0 ‚Üí value; refresh animates previous ‚Üí new value.

**Frames.** Route `progress.FrameMsg` to every bar's `Update` inside the model's
`Update`, batching the resulting cmds. Tune `SetSpringOptions` for a subtle,
quick settle (explicit goal: not a noisy dashboard).

**Invariants (the cases that must not break animation):**

- Hidden provider ‚Üí row not rendered and not animated.
- Missing data ‚Üí bar idle at 0, placeholder row ("‚Äî") shown as today.
- Stale data ‚Üí no special animation behavior (freshness line already conveys
  staleness).
- Resize (`tea.WindowSizeMsg`) ‚Üí `SetWidth` only; never restarts an animation.

## Data Flow

1. `main.go` parses flags ‚Üí builds `DisplayPrefs` ‚Üí `tui.NewModel(..., WithDisplayPrefs(prefs))`.
2. `Model.Init` kicks the first refresh + tick (unchanged).
3. `refreshMsg` ‚Üí `mergeRefresh` updates windows; new/changed targets trigger
   `SetPercent` cmds ‚Üí batched out of `Update`.
4. `progress.FrameMsg` ‚Üí fan out to each bar's `Update` ‚Üí batch follow-up frame
   cmds until `IsAnimating()` is false.
5. `KeyPressMsg` `b`/`v` mutate `DisplayPrefs` in the model (no refresh needed).
6. `View`/`render` reads prefs: filters rows by visibility, sets fill rune by bar
   style, and renders each visible bar via its persistent model.

## Error Handling & Edge Cases

- Invalid/unknown CLI flags: usage to stderr, exit 2. No partial application.
- `--only` with a hidden-but-erroring source: footer recovery hints follow
  visibility (a hidden provider shows no hint).
- Animation never blocks data: a runaway frame loop is avoided because Bubbles
  stops emitting frames once the bar settles (`IsAnimating()` returns false).
- Refresh failure preserves last-known-good rows (existing behavior in
  `mergeRefresh`); their bars stay at the last animated value, no re-animation.

## Testing Strategy (TEST-01, TEST-02)

Automated only. Extend existing table/width-driven tests.

- **`internal/tui/view_test.go`**
  - Segmented vs solid: assert the rendered fill rune (`‚ñå` vs `‚ñà`) for a known
    percentage.
  - Provider visibility: Claude-only, Codex-only, and both layouts render the
    correct rows + freshness lines at all six widths (80, 50, 49, 30, 29, 20).
  - Hidden provider suppresses its freshness line and footer hint.
- **`internal/tui/update_test.go`**
  - `b` toggles bar style; `v` cycles visibility through the three states and
    back.
  - `SetPercent` is invoked with the correct target on initial data and on a
    value-changing refresh (assert via `Percent()` after settle / via emitted
    cmd presence).
  - `FrameMsg` advances animation and it eventually settles
    (`IsAnimating()` true ‚Üí false).
  - Hidden / missing / resize events produce no panic and no stray animation of
    non-visible or absent rows.
- **`cmd/llm-quota/main_test.go`**
  - Valid flags produce the expected `DisplayPrefs`.
  - Invalid `--only` value and unknown flag ‚Üí exit 2.
  - `--help` ‚Üí usage text, exit 0.

**Determinism note:** the spring is wall-clock timed, so tests assert
*orchestration* ‚Äî which targets are set, which cmds are emitted, and
`IsAnimating()` transitions ‚Äî not exact intermediate frame pixels.

**Verification commands:** `go build ./...`, `go test ./...`, `go test -race ./...`, `go vet ./...`.

## Out of Scope / Accepted Gaps

- **Real tmux-pane human UAT (UAT-01):** intentionally not performed; covered by
  automated width/render tests instead. Real-pane subjective "feel" is an
  accepted gap.
- No new data sources, networking, OAuth, or credential reads.
- No persistence of preferences across runs (no config file) ‚Äî set per launch
  via flags, adjust live via keys.
- No alerts, notifications, history, or forecasting.

## Success Criteria

Phase 8:

1. TUI runs with either the segmented (default) or solid bar style.
2. A Claude-only view omits Codex rows; a Codex-only view omits Claude rows.
3. No preference combination can hide both sources (structural).
4. Bar style and visibility are settable via CLI flags and toggleable at runtime
   (`b`, `v`).
5. README and `--help` document the options.

Phase 9:

1. Initial load animates visible bars from empty to current usage.
2. Manual refresh (`r`) animates visible bars to the refreshed usage.
3. Automatic refresh animates visible bars from prior to refreshed value.
4. Hidden providers, missing data, stale data, and resize do not produce broken
   animation states.
5. Deterministic tests cover alignment, freshness, provider visibility, bar
   modes, and animation state transitions.

## Build Order

1. **Foundation:** persistent per-row `progress.Model` (no behavior change).
2. **Phase 8:** prefs types + `WithDisplayPrefs` + flags + `b`/`v` toggles +
   visibility filtering + solid-bar rune + README/`--help`.
3. **Phase 9:** animation drive (`SetPercent`) + `FrameMsg` routing + spring
   tuning + invariants.
4. **Tests:** broaden deterministic coverage across bar modes, visibility, and
   animation transitions.
